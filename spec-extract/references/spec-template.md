# Spec Template

Use this as the target structure for the finished document.

During an active walkthrough, write only the header plus sections that the developer has already approved. Do not pre-populate untouched sections with placeholders or speculative text.

```markdown
# Spec: {Workflow Name}

> Generated by spec-extract on {date}
> Status: DRAFT | REVIEWED
> Entry point: {file:function}
> Reviewed by: {developer name or "pending"}

## 1. Purpose
What this workflow does in 1-3 sentences.
Write this from the developer's confirmed wording, not from agent inference.

## 2. Entry Point
Where this workflow starts.
Include file, function, trigger mechanism, and the agreed boundary.

## 3. Inputs
What data comes in.
Include parameters, request body, event payload, CLI args, or config inputs.
Include validation rules only when confirmed from code or by the developer.

## 4. Step-by-Step Flow
Numbered sequence of what happens.
Reference `file:function` or `file:line` for each step.
Include branching logic only after the developer confirms it belongs in scope.

## 5. External Interactions
Every call to something outside this codebase:
DB reads/writes, API calls, queues, caches, files, or other services.
For each interaction capture service name, operation type, and citation.

## 6. Side Effects
Things this workflow causes beyond its return value.
For each side effect capture what happens, when it triggers, and its citation.

## 7. Error Handling
How failures are handled at each step.
Capture retries, swallowed errors, propagated errors, fallbacks, compensating actions, and timeouts.

## 8. Edge Cases & Implicit Assumptions
Capture surprising branches, unvalidated assumptions, hard-coded values, null handling, timezone/currency assumptions, and similar risks.
Flag how each item was discovered.

## 9. Behavior Classification
Record every non-trivial confirmed behavior.

| Behavior | Location | Classification | Notes |
|----------|----------|----------------|-------|
| {behavior} | {file:line or file:function} | {tag} | {developer-confirmed context} |

Allowed tags:
- REQUIRED
- ALLOWED_UNDESIRABLE
- DEPRECATED
- BUG_COMPATIBLE
- DEAD_PATH
- UNKNOWN

## 10. Open Questions
Capture unresolved items.
For each item include what is unclear, where in code it surfaced, and why it matters.

## 11. Decisions (ADRs)
Link ADRs created during this session.

Format:
- [ADR-{NNN}: {Title}](../decisions/{NNN}-{slug}.md)
```

## Writing Notes

- Keep citations inside each section rather than as an appendix.
- Prefer direct, present-tense language.
- Keep confirmed facts in the main sections and uncertainty in Open Questions.
- Preserve the developer's corrections even when the code appears to disagree.
